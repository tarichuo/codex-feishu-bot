# Feishu Codex Bot Requirements

## 1. 概述

本项目需要实现一个 Python 编写的飞书机器人服务端。服务端通过飞书提供的长连接机制接收和发送消息，并将用户消息转发到本机运行的 Codex app server。Codex app server 采用 JSON-RPC 协议，其接口定义以仓库 `schemas/` 目录为准。

系统需要同时支持单聊和群聊两种对话模式，支持文本、图片、流式回复、slash 命令。飞书应用由用户自行在飞书开放平台创建，服务端通过环境变量读取单个机器人实例的配置；如需运行多个机器人，由用户启动多个服务进程分别承载。

## 2. 目标

- 提供一个可运行的飞书机器人服务端。
- 支持通过多次启动服务进程的方式运行多个机器人实例，每个进程绑定一个独立的飞书应用配置。
- 支持将飞书消息可靠转发到本机 Codex app server，并将 Codex 的输出回传到飞书。
- 保证单聊和群聊具有明确且可预测的 thread 复用策略。
- 支持流式输出、图片收发、slash 命令和处理中表情反馈。
- 输出一份符合飞书开放平台要求的权限配置文档，供用户直接复制粘贴到飞书开放平台网页端。
- 运行时应产出清晰日志，便于问题排查和调试。
- 只允许预先配置的合法用户与机器人对话，防止未授权用户借由机器人操作本机 Codex。

## 3. 非目标

- 不负责替用户在飞书开放平台中创建机器人应用。
- 不负责托管远程 Codex 服务，默认只连接本机 Codex app server。
- 当前需求未要求实现多租户权限系统、消息审计后台或 Web 管理界面。

## 4. 术语

- 机器人实例：服务端中的一个运行单元，对应一个飞书机器人应用配置。
- 单聊会话：飞书用户与某个机器人实例的一对一聊天上下文。
- 群聊会话：机器人实例在某个群中的聊天上下文。
- Thread：Codex app server 中的对话线程标识，用于持续多轮上下文。
- Slash 命令：以 `/` 开头的用户输入，按命令语义转发给 Codex。

## 5. 用户故事

### 5.1 多实例机器人

- 作为部署者，我希望通过启动多个服务进程的方式运行多个机器人实例，以便每个进程承载一个飞书机器人。
- 作为部署者，我希望每个机器人实例都通过独立环境变量配置自己的 app id、app secret、长连接凭据和本机 Codex 连接信息。

### 5.2 单聊对话

- 作为飞书用户，我希望在单聊中直接向机器人发送消息，并获得 Codex 的回复。
- 作为飞书用户，我希望在 1 小时内继续提问时复用上一次 Codex thread，以保持上下文连续。
- 作为飞书用户，我希望在距离上次对话超过 1 小时时自动开始新的 Codex thread，避免旧上下文干扰。

### 5.3 群聊对话

- 作为群成员，我希望机器人被拉入群后就为该群建立一个独立的 Codex thread。
- 作为群成员，我希望只有 `@机器人` 的消息才会被发送给 Codex，避免机器人响应无关群消息。
- 作为群成员，我希望群聊 thread 在群生命周期内长期保留，不因时间过期或服务重启而丢失，除非用户主动执行 `/clear`。

### 5.4 Slash 命令

- 作为用户，我希望发送 `/clear`、`/model`、`/compact`、`/init` 等命令时，机器人能将其识别为 slash 命令并转发给 Codex。
- 作为用户，我希望在群聊里 `@机器人` 后发送 slash 命令时，机器人同样能识别并处理。
- 作为用户，我希望发送未支持的 slash 命令时，机器人明确提示当前支持的命令列表，而不是静默失败。
- 作为用户，我希望 slash 命令与普通消息走统一的回复链路，从而同样获得流式反馈。

### 5.5 流式回复与处理中状态

- 作为用户，我希望机器人回复以流式方式持续更新，而不是长时间静默后一次性返回。
- 作为用户，我希望针对 Codex 的一次回复，飞书侧始终更新同一条回复消息；当我开始下一轮对话时，再创建新的回复消息。
- 作为用户，我希望机器人收到消息后立刻给原消息添加“敲键盘”表情，以表示正在处理中。
- 作为用户，我希望在机器人回复完成或失败后，该“敲键盘”表情被取消。

### 5.6 图片收发

- 作为用户，我希望向机器人发送图片，并由机器人将图片内容按可支持的方式转发给 Codex。
- 作为用户，我希望机器人先将飞书图片下载到本地，再把本地图片路径传递给 Codex。
- 作为用户，我希望当 Codex 返回图片内容时，机器人能够将图片发送回飞书。
- 作为用户，我希望机器人支持多图输入以及图文混排消息。

### 5.7 平台权限与配置

- 作为部署者，我希望获得一份符合飞书平台导入要求的权限配置文档，以便直接复制内容并粘贴到飞书开放平台网页端。
- 作为部署者，我希望所有敏感配置通过环境变量提供，而不是写死在代码中。
- 作为部署者，我希望服务在运行时输出结构清晰、可定位问题的日志，便于调试飞书接入、消息路由和 Codex 交互问题。
- 作为部署者，我希望通过环境变量配置允许对话的飞书用户白名单，只有这些用户才能驱动机器人与 Codex 交互。
- 作为部署者，我希望当非白名单用户尝试与机器人对话时，系统自动拦截，并通过单聊向 owner 发送告警。

## 6. 功能需求

### FR-1 多实例启动

- 系统应以单进程单机器人实例的方式运行。
- 系统应允许部署者通过多次启动本项目来运行多个机器人实例。
- 系统应允许每个服务进程绑定一套独立的飞书应用凭据和运行配置。

### FR-2 飞书长连接接入

- 系统应使用飞书长连接机制与飞书后台通信。
- 系统应接收来自飞书的消息事件、群成员事件或建立群聊上下文所需的事件。
- 系统应支持向飞书发送文本、图片、表情状态相关操作以及流式消息更新。

### FR-3 Codex app server 接入

- 系统应连接本机的 Codex app server。
- 系统与 Codex app server 的交互应使用 JSON-RPC。
- 系统实现时应以仓库 `schemas/` 目录为协议依据。
- 系统应支持创建 thread、向指定 thread 发送消息、接收流式输出，以及处理 slash 命令相关请求。

### FR-4 单聊会话路由

- 当系统收到单聊消息时，应定位该用户在该机器人实例下的最近一次单聊会话。
- 单聊会话的复用范围应按“同一用户 + 同一机器人实例”维度计算。
- 当最近一次单聊消息时间距当前时间超过 1 小时时，系统应创建新的 Codex thread。
- 当最近一次单聊消息时间距当前时间不超过 1 小时时，系统应复用已有 Codex thread。
- 单聊会话应至少记录用户标识、机器人实例标识、最近活跃时间和 thread 标识。

### FR-5 群聊会话路由

- 当机器人被加入某个群聊后，系统应为该群建立一个 Codex thread。
- 当群成员发送 `@机器人` 的消息时，系统应将该消息发送到该群对应的 Codex thread。
- 对于未 `@机器人` 的群消息，系统默认不发送给 Codex。
- 群聊 thread 不应因空闲时间过期。
- 群聊 thread 不应因服务重启而丢失。
- 只有当用户主动发送 `/clear` 时，系统才应为该群重建 thread。
- 当机器人被移出群聊后再次拉入时，系统应为该群创建新的 thread。

### FR-6 Slash 命令处理

- 当收到以 `/` 开头的消息时，系统应将其识别为 slash 命令。
- 当群聊消息以 `@机器人` 开头且正文中的有效指令以 `/` 开头时，系统也应将其识别为 slash 命令。
- 系统当前至少支持 `/clear`、`/model`、`/compact`、`/init`。
- 当 slash 命令属于支持列表时，系统应将其原样透传到 Codex，而不是在飞书侧本地解释其业务语义。
- 当 slash 命令不在支持列表中时，系统应向用户返回错误提示，内容包括“不支持的slash命令”以及当前支持的命令列表。
- `/clear` 的语义由 Codex 决定，飞书侧仅负责透传和显示结果。
- `/model` 允许携带参数，例如 `/model gpt-5.4`，参数解析与实际处理由 Codex 决定。

### FR-7 流式输出

- 当 Codex 返回流式内容时，系统应将其增量同步到飞书。
- 当飞书流式消息能力存在长度、频率或消息类型限制时，系统应在实现中兼容这些限制。
- 对于 Codex 产生的一次回复，系统应在飞书侧持续更新同一条回复消息。
- 当用户发起下一条对话时，系统应为新的 Codex 回复创建新的飞书消息。
- 当流式回复结束时，系统应将最终状态收敛为完整可读的飞书消息。

### FR-8 处理中表情

- 系统在收到用户消息后，应尽快在该消息上添加“敲键盘”表情。
- 系统在回复成功结束、失败结束或被中断时，应移除该表情。

### FR-9 图片接收与发送

- 系统应支持接收飞书用户发送的图片消息。
- 系统应支持接收包含多张图片的消息输入。
- 系统应支持包含文本与图片混排的消息输入。
- 系统应先将飞书图片下载到本地文件系统。
- 系统应将本地图片路径按 Codex app server 可接受的方式转发给 Codex，并保持与原始文本内容的顺序关系。
- 当 Codex 输出图片时，系统应支持将图片发送到飞书。
- 当 Codex 输出多张图片或图文混排内容时，系统应尽可能在飞书中保留其表达顺序与可读性。
- 如果飞书图片消息与 Codex 图片输入输出格式不一致，系统应负责格式转换。

### FR-10 配置与权限文档

- 系统应通过环境变量读取运行配置。
- 系统应提供一份飞书开放平台权限需求文档。
- 权限文档应列出机器人运行所需的事件订阅、消息能力、图片能力和群聊相关权限。
- 权限文档的内容格式应符合飞书开放平台网页端可直接粘贴使用的要求。
- 系统应通过环境变量配置允许对话的飞书用户 ID 白名单。
- 系统应通过环境变量配置告警接收者 owner 的飞书用户 ID。

### FR-11 会话状态持久化

- 系统应持久化保存群聊与单聊的 thread 映射和必要元数据。
- 系统重启后应能够恢复既有群聊 thread，并继续使用原有 thread。
- 对于单聊会话，系统重启后若仍处于 1 小时有效期内，应能够继续复用原 thread。

### FR-12 Codex 输出类型映射

- 系统应支持将 Codex 的文本输出映射为飞书文本消息。
- 系统应支持将 Codex 的图片输出映射为飞书图片消息。
- 系统应支持将 Codex 的代码块输出以适合飞书展示的文本格式发送给用户。
- 当 Codex 输出文件时，系统应将文件直接发送给用户。
- 当 Codex 输出审批请求时，系统应调用飞书机器人可用的审批消息相关 API 向用户发送审批消息。
- 对于 Codex 支持但飞书侧暂不支持直接映射的消息类型，系统应在设计阶段定义降级展示策略。

### FR-13 消息幂等与去重

- 系统应保证同一条飞书用户消息最多只向 Codex 转发一次。
- 系统应基于飞书事件标识、消息标识或其他可用唯一键实现幂等控制。
- 当飞书因重试、重连或重复投递再次发送同一条消息事件时，系统不应重复创建 Codex 输入或重复发送飞书回复。
- 幂等记录应具备合理的持久化或生存期策略，以覆盖服务重启和短期重复投递场景。

### FR-14 运行时日志

- 系统应输出清晰、可检索的运行时日志，用于调试和问题排查。
- 日志应覆盖至少以下关键链路：飞书长连接建立与断开、事件接收、消息去重判定、thread 选择或创建、Codex 请求发送、流式回复过程、图片下载与发送、审批消息发送、错误与重试。
- 日志应包含必要的上下文标识，例如机器人实例、会话类型、用户或群标识、飞书消息标识、Codex thread 标识和请求阶段。
- 日志不应泄露敏感凭据；涉及密钥、token、图片二进制内容时应脱敏或避免直接输出。

### FR-15 用户白名单访问控制

- 系统应在处理任何用户消息前，先校验消息发送者的飞书用户 ID 是否存在于配置的白名单中。
- 只有白名单内用户的消息才允许进入后续链路，包括 slash 命令、文本、图片、图文混排、单聊和群聊 `@机器人` 消息。
- 对于非白名单用户的消息，系统不得将消息内容转发给 Codex，不得创建或推进任何 Codex thread。
- 对于非白名单用户的消息，系统应直接过滤，不向该用户返回业务回复。
- 对于非白名单用户的消息，系统应通过单聊向 owner 发送告警消息。
- 告警消息应至少包含：触发时间、发送者用户 ID、会话类型、群聊 ID 或单聊标识、原始飞书消息 ID。
- 白名单拦截事件应写入运行日志。

## 7. 验收标准

### AC-1 多实例

- 当部署者分别使用两套配置启动两个服务进程时，系统能够同时连接两个机器人实例，且消息不会串线。

### AC-2 单聊 thread 复用

- 当同一用户在 1 小时内连续两次向同一机器人发送消息时，系统使用相同 thread。
- 当同一用户第二次发消息距离第一次超过 1 小时时，系统使用新 thread。
- 当同一用户在其他机器人实例中发起会话时，不应复用当前机器人实例下的 thread。

### AC-3 群聊 thread 生命周期

- 当机器人被加入一个新群时，系统创建该群专属 thread。
- 当群内用户发送 `@机器人` 消息时，系统将消息发送到该群 thread。
- 当群内用户发送未 `@机器人` 的消息时，系统不向 Codex 转发。
- 当群闲置超过 1 小时后再次 `@机器人` 时，系统仍复用原 thread。
- 当服务重启后，群内再次 `@机器人` 时，系统仍复用原 thread。
- 当群内用户发送 `/clear` 后，系统清理当前群 thread 并创建新 thread。
- 当机器人被移出群后重新拉入同一群时，系统创建新的群聊 thread。

### AC-4 Slash 命令

- 当用户发送 `/clear`、`/model`、`/compact`、`/init` 时，系统将其作为 slash 命令转发至 Codex 并返回结果。
- 当群内用户发送 `@机器人 /clear` 等受支持命令时，系统同样将其作为 slash 命令处理。
- 当用户发送未支持的 slash 命令时，系统返回“不支持的slash命令，当前支持的命令包括 /clear /model /compact /init”。
- 当用户发送 `/model gpt-5.4` 时，系统将完整命令透传给 Codex。

### AC-5 流式回复

- 当 Codex 产生分段输出时，飞书侧能看到同一条消息被连续更新，而不是只在结束时返回一次。
- 当用户发起下一条对话时，系统创建新的飞书回复消息，而不是覆盖上一轮回复。

### AC-6 处理中表情

- 当用户发送消息后，原消息出现“敲键盘”表情。
- 当机器人回复结束后，该表情被取消。

### AC-7 图片能力

- 当用户发送图片时，系统能够接收并转发给 Codex。
- 当用户发送多张图片或图文混排消息时，系统能够按原始顺序组织内容并转发给 Codex。
- 当 Codex 返回图片时，系统能够在飞书中发送该图片。
- 当 Codex 返回多张图片或图文混排内容时，系统能够在飞书中以可读方式发送给用户。

### AC-8 配置与文档

- 当部署者阅读权限文档后，能够在飞书开放平台完成机器人权限配置。
- 当部署者设置环境变量后，服务无需修改源码即可启动目标机器人实例。

### AC-9 会话持久化

- 当服务重启后，同一群聊的后续消息仍命中重启前的 thread。
- 当服务重启后，仍在 1 小时有效期内的单聊会话能够继续复用重启前的 thread。

### AC-10 Codex 输出类型映射

- 当 Codex 返回代码块时，用户能在飞书中看到格式可读的代码内容。
- 当 Codex 返回文件时，用户能在飞书中直接收到该文件。
- 当 Codex 返回审批请求时，系统能够通过飞书审批消息能力向用户发送审批消息。

### AC-11 消息幂等与去重

- 当飞书重复投递同一条用户消息事件时，系统只向 Codex 转发一次该消息。
- 当飞书重复投递同一条用户消息事件时，用户在飞书侧不会看到重复的机器人回复。

### AC-12 运行时日志

- 当服务正常处理一条消息时，日志中能够串联看到飞书事件接收、幂等判定、thread 选择、Codex 请求和飞书回复等关键步骤。
- 当图片下载、Codex 调用或飞书发送失败时，日志中能够定位失败阶段和相关上下文标识。
- 日志中不出现 app secret、access token 等敏感配置明文。

### AC-13 用户白名单访问控制

- 当白名单用户发送单聊消息或群聊 `@机器人` 消息时，系统正常进入会话路由和 Codex 处理链路。
- 当非白名单用户发送消息时，系统不会向 Codex 转发该消息，也不会创建或更新对应 thread。
- 当非白名单用户发送消息时，owner 会收到一条单聊告警消息。
- 当非白名单用户重复触发时，日志中能看到对应的拦截记录和告警发送结果。

## 8. 约束与假设

- 代码实现语言固定为 Python。
- Codex app server 运行在本机，且服务端可通过本地地址访问。
